# 流程定义使用说明

> 适用范围：RuoYi-Vue BPM v2（`ruoyi-bpm-v2`）流程管理模块  
> 文档版本：v1.0  
> 最后更新：2026-06-27

---

## 1. 概述

### 1.1 什么是流程定义

**流程定义**是对企业业务流程的数字化描述，它规定了：

- 流程由哪些节点（审批、抄送、分支等）组成；
- 节点之间的流转顺序；
- 每个节点由谁处理、需要填写哪些字段、能否退回或转办；
- 流程在什么条件下走哪条分支。

在本系统中，流程定义是 BPM v2 自定义引擎的核心配置入口。一个流程定义发布后即可被用户启动，生成**流程实例**和待办**任务**。

### 1.2 核心概念

| 概念 | 说明 | 对应数据表 |
|---|---|---|
| 流程定义（Definition） | 流程的元数据，包含流程 Key、名称、分类、状态、最新版本号等 | `bpm_process_definition` |
| 流程版本（Version） | 每个定义下的一个可发布版本，包含 BPMN XML、变更说明、发布状态 | `bpm_definition_version` |
| 流程实例（Instance） | 流程定义被启动后产生的运行中记录 | `bpm_process_instance` |
| 任务（Task） | 流程运行到某个节点时，需要用户处理的待办事项 | `bpm_task` |
| 分类（Category） | 对流程定义进行分组管理，支持树形结构 | `bpm_category` |
| 表单（Form） | 绑定到流程或节点的表单 Schema | `bpm_form_definition` |
| 字段权限 | 控制节点上各字段的可见/可编辑/隐藏/必填状态 | `bpm_field_permission` |
| 变量（Variable） | 流程运行时可使用的动态数据，可用于条件表达式 | `bpm_variable` |

### 1.3 界面入口

登录系统后，通过菜单进入：

```
流程管理 → 流程定义
```

页面左侧为**流程分类树**，右侧为**流程定义列表**。

## 2. 流程定义列表管理

### 2.1 查看流程定义列表

进入【流程定义】页面后，系统默认展示全部流程定义。您可以通过以下方式筛选：

- **分类筛选**：点击左侧分类树节点，仅显示该分类下的流程定义；
- **关键字搜索**：在右上角搜索框输入流程名称或流程 Key；
- **状态筛选**：按草稿/已发布/已停用状态过滤。

列表字段说明：

| 字段 | 说明 |
|---|---|
| 流程名称 | 流程定义的显示名称 |
| 流程 Key | 流程唯一标识，启动流程时使用，创建后不可修改 |
| 分类 | 所属流程分类 |
| 状态 | 草稿 / 已发布 / 已停用 |
| 最新版本 | 当前已发布的最新版本号 |
| 创建时间 | 流程定义创建时间 |
| 操作 | 设计、发布、编辑、删除、版本管理等入口 |

### 2.2 新增流程定义

**使用场景**：需要上线一个新的审批或业务流程。

**操作步骤**：

1. 点击页面右上角【新增】按钮；
2. 在弹窗中填写：
   - **流程名称**：必填，建议简洁明了，如"请假申请流程"；
   - **流程 Key**：必填，全局唯一，建议英文+数字+下划线，如 `leave_apply`；
   - **分类**：选择所属分类，便于后续管理；
   - **备注**：可选，补充说明流程用途。
3. 点击【确定】。

**注意事项**：

- 流程 Key 创建后不可修改，请谨慎命名；
- 新增成功后，系统会自动生成一个初始草稿版本 `1.0`。

### 2.3 编辑基础信息

**使用场景**：修改流程名称、所属分类或备注，不影响流程版本内容。

**操作步骤**：

1. 在列表中找到目标流程定义，点击操作列【编辑】；
2. 修改基础信息（流程 Key 不可编辑）；
3. 点击【确定】保存。

### 2.4 删除流程定义

**使用场景**：流程已废弃且没有运行中的实例时，彻底删除。

**操作步骤**：

1. 在列表中选择一条或多条流程定义；
2. 点击【删除】按钮；
3. 确认删除。

**注意事项**：

- 删除流程定义会同时删除其所有历史版本；
- 如果该流程定义下存在运行中或已完成的流程实例，请谨慎操作，避免历史数据无法追溯。

### 2.5 启用 / 停用流程定义

**使用场景**：

- **停用**：流程需要临时下线，不允许再启动新流程实例；
- **启用**：恢复流程可用状态。

**操作步骤**：

1. 在列表中找到目标流程定义；
2. 点击操作列【停用】或【启用】按钮；
3. 确认操作。

**注意事项**：

- 停用只影响新启动的流程，已运行中的流程实例不受影响；
- 只有已发布的流程定义才能停用，草稿状态无法停用。

## 3. 流程模型设计器

### 3.1 进入设计器

**操作步骤**：

1. 在流程定义列表中，点击目标流程的【设计】按钮；
2. 系统打开 BPMN 流程设计器页面，左侧为元素面板，中间为画布，右侧为属性面板。

### 3.2 画布基本操作

| 操作 | 说明 |
|---|---|
| 拖拽节点 | 从左侧工具面板拖拽节点到画布 |
| 连接节点 | 选中源节点，点击出现的箭头并拖拽到目标节点 |
| 选中元素 | 单击节点或连线，右侧显示属性面板 |
| 删除元素 | 选中后按 Delete 键或点击删除图标 |
| 平移画布 | 按住空白处拖拽 |
| 缩放画布 | 使用右下角缩放控件或鼠标滚轮 |

### 3.3 支持的节点类型

| 图标 | 节点类型 | 用途 |
|---|---|---|
| 圆形 | 开始事件（Start Event） | 流程起点，每个流程必须有且仅有一个 |
| 圆角矩形 | 用户任务（User Task） | 需要人工审批或处理的节点 |
| 菱形 | 排他网关（Exclusive Gateway） | 根据条件判断走向不同分支 |
| 圆形+叉 | 并行网关（Parallel Gateway） | 同时进入多个分支或等待多个分支汇合 |
| 粗线圆形 | 结束事件（End Event） | 流程终点 |

### 3.4 绘制标准审批流程

以"请假申请"为例，标准流程为：

```
开始 → 提交申请（发起人填写） → 部门经理审批 → 人事审批 → 结束
```

**操作步骤**：

1. 拖入一个【开始事件】到画布；
2. 拖入一个【用户任务】作为"提交申请"节点；
3. 从开始事件连接一条线到"提交申请"；
4. 继续拖入"部门经理审批"和"人事审批"两个用户任务；
5. 依次连接；
6. 拖入【结束事件】并连接到最后一个节点。

### 3.5 配置节点通用属性

选中任意节点后，右侧属性面板可配置：

| 属性 | 说明 |
|---|---|
| ID | 节点唯一标识，建议保持默认或按规范命名 |
| 名称 | 节点显示名称，如"部门经理审批" |
| 文档 | 可选，补充节点说明 |

**建议命名规范**：

- 使用中文名称，便于业务人员理解；
- 同一流程内节点名称不要重复，避免审批记录混淆。

## 4. 办理人 / 候选人配置

### 4.1 配置场景总览

流程节点需要明确"由谁处理"。系统支持以下配置方式：

| 配置方式 | 使用场景 | 配置位置 |
|---|---|---|
| 指定用户 | 节点固定由某几个人审批 | 属性面板 → 办理人 |
| 指定角色 | 节点由拥有某角色的用户审批，如"部门经理" | 属性面板 → 候选角色 |
| 指定部门 | 节点由某部门下的用户审批，如"人事部" | 属性面板 → 候选部门 |
| 发起人 | 节点回到流程发起人自己处理 | 属性面板 → 办理人类型 |
| 表达式 | 根据流程变量动态计算办理人 | 属性面板 → 表达式 |

### 4.2 指定用户办理

**使用场景**：如总经理审批节点固定由特定人员处理。

**操作步骤**：

1. 选中用户任务节点；
2. 在右侧属性面板找到【办理人】或【候选人】配置项；
3. 点击选择用户，可单选或多选；
4. 保存。

**区别说明**：

- **办理人（Assignee）**：任务只分配给这一个人；
- **候选人（Candidate Users）**：任务出现在多个候选人的待办列表中，谁先拾取谁处理。

### 4.3 指定角色办理

**使用场景**：如"部门经理审批"节点，不固定某个人，而是所有具有"部门经理"角色的人都能看到待办。

**操作步骤**：

1. 选中用户任务节点；
2. 在属性面板找到【候选组】配置项；
3. 选择角色；
4. 保存。

**运行效果**：所有属于该角色的用户都会在自己的待办列表中看到该任务。

### 4.4 指定部门办理

**使用场景**：如"人事部审批"节点，由人事部所有成员处理。

**操作步骤**：

1. 选中用户任务节点；
2. 在属性面板找到【候选部门】配置项；
3. 选择部门；
4. 保存。

### 4.5 发起人办理

**使用场景**：流程中某个节点需要回到发起人处补充材料或确认。

**操作步骤**：

1. 选中用户任务节点；
2. 在属性面板中将办理人类型选择为【发起人】；
3. 保存。

### 4.6 表达式办理

**使用场景**：办理人需要根据流程变量动态确定，如"申请人的直属上级"。

**配置方式**：

1. 在属性面板中选择办理人类型为【表达式】；
2. 输入 SpEL 表达式，例如：
   ```
   ${directManagerId}
   ```
3. 确保流程启动或上游节点已将 `directManagerId` 变量传入。

**常见变量**：

| 变量名 | 说明 |
|---|---|
| `starter` | 流程发起人用户 ID，系统自动注入 |
| `approvalAssignee` | 启动时指定的首节点办理人 |
| 自定义变量 | 在流程变量或表单中定义 |

## 5. 会签、或签与依次审批

### 5.1 审批模式说明

对于需要多人共同审批的节点，系统支持三种模式：

| 模式 | 说明 | 使用场景 |
|---|---|---|
| 会签（COUNTER_SIGN） | 所有办理人都必须同意，节点才算通过 | 重大事项需全员通过 |
| 或签（ANY_SIGN） | 任意一个办理人同意，节点即通过 | 普通审批，多人中任一处理即可 |
| 依次审批（SEQUENTIAL） | 按顺序依次审批，所有人都需通过 | 逐级审批，如部门经理 → 总监 → 总经理 |

### 5.2 配置多实例审批

**操作步骤**：

1. 选中需要配置会签/或签的用户任务节点；
2. 在右侧属性面板中找到【多实例】配置；
3. 选择审批模式：
   - **会签**：所有实例完成后才继续；
   - **或签**：任一实例完成即继续；
   - **依次审批**：按集合顺序依次创建任务。
4. 配置集合变量（Collection），例如：
   ```
   ${approverList}
   ```
5. 配置元素变量（Element Variable），例如：
   ```
   assignee
   ```
6. 在节点办理人表达式中引用元素变量，例如：
   ```
   ${assignee}
   ```

**运行说明**：

- 启动流程时传入 `approverList` 变量，值为用户 ID 列表；
- 系统会为列表中每个用户创建一个任务实例；
- 会签模式下，所有人都审批通过才进入下一节点；
- 或签模式下，只要有一人审批通过就进入下一节点。

## 6. 条件表达式与分支配置

### 6.1 使用场景

当流程需要根据业务条件走向不同分支时，使用**排他网关（Exclusive Gateway）**和**条件表达式**。

例如请假流程：

- 请假天数 ≤ 3 天：部门经理审批即可；
- 请假天数 > 3 天：需要部门经理和总监两级审批。

### 6.2 配置条件分支

**操作步骤**：

1. 在画布中拖入一个【排他网关】；
2. 从网关引出两条连线到不同节点；
3. 选中第一条连线，在右侧属性面板中设置条件表达式，例如：
   ```
   ${leaveDays <= 3}
   ```
4. 选中第二条连线，设置条件表达式：
   ```
   ${leaveDays > 3}
   ```
5. 保存。

### 6.3 条件表达式语法

系统使用 SpEL 表达式，支持以下写法：

| 示例 | 含义 |
|---|---|
| `${leaveDays > 3}` | 请假天数大于 3 |
| `${amount >= 10000}` | 金额大于等于 10000 |
| `${type == 'urgent'}` | 类型为紧急 |
| `${leaveDays <= 3 && amount < 5000}` | 多个条件同时满足 |
| `${type == 'A' \|\| type == 'B'}` | 多个条件任一满足 |

**注意事项**：

- 表达式中使用的变量必须先在流程变量或表单中定义；
- 字符串比较使用单引号；
- 排他网关中必须有一条默认分支（无条件或兜底条件），避免流程卡死。

## 7. 流程版本管理

### 7.1 版本概念

每个流程定义可以有多个版本，版本号格式为 `主版本.次版本`，例如 `1.0`、`1.1`、`2.0`。

| 版本状态 | 说明 |
|---|---|
| 草稿（0） | 正在编辑中的版本，不可启动流程 |
| 已发布（1） | 可正常使用的版本 |
| 已停用（2） | 不再允许启动，但历史实例仍可查询 |

### 7.2 保存草稿

**操作步骤**：

1. 在设计器中完成流程绘制和配置；
2. 点击工具栏【保存草稿】按钮；
3. 系统会保存当前 BPMN XML 到最新草稿版本。

**版本生成规则**：

- 如果当前已有草稿版本，直接覆盖该草稿；
- 如果没有草稿版本，系统自动递增版本号并创建新草稿。

### 7.3 发布流程

**使用场景**：流程设计完成并经过测试后，正式发布供业务使用。

**操作步骤**：

1. 在流程设计器中点击【发布】按钮；或在流程定义列表中点击【发布】；
2. 系统会对 BPMN XML 进行校验，检查：
   - XML 是否为空；
   - 是否包含 `<definitions>` 根节点；
   - 是否有开始事件；
   - 是否有结束事件。
3. 校验通过后，版本状态变为【已发布】；
4. 流程定义的【最新版本】字段同步更新。

**注意事项**：

- 发布后的流程版本不能直接编辑；
- 如果需要修改，系统会自动基于已发布版本创建新的草稿版本。

### 7.4 查看历史版本

**操作步骤**：

1. 在流程定义列表中，点击目标流程的【版本管理】按钮；
2. 系统弹出该流程的所有历史版本列表；
3. 可查看每个版本的版本号、状态、发布人、发布时间、变更说明。

### 7.5 切换 / 回滚版本

**使用场景**：新发布的流程版本出现问题，需要回退到之前的稳定版本。

**操作步骤**：

1. 在版本管理列表中，找到目标历史版本；
2. 点击【设为最新版本】或【回滚】按钮；
3. 确认操作。

**注意事项**：

- 回滚后，新启动的流程实例将使用回滚后的版本；
- 已经运行的流程实例仍使用原版本继续执行。

### 7.6 停用版本

**操作步骤**：

1. 在版本管理列表中，找到目标版本；
2. 点击【停用】按钮；
3. 确认操作。

## 8. 表单与字段权限配置

### 8.1 绑定表单

**使用场景**：为流程定义或特定节点配置需要填写的表单。

**操作步骤**：

1. 进入流程设计器或流程定义详情页；
2. 找到【表单配置】入口；
3. 选择或新建一个表单定义；
4. 设置表单作用范围：
   - **全局表单**：整个流程共用；
   - **节点表单**：仅当前节点使用。

### 8.2 字段权限配置

**使用场景**：控制不同节点上各字段的显示和编辑权限。例如：

- 发起节点：所有字段可编辑；
- 审批节点：只能填写审批意见，其他字段只读；
- 归档节点：所有字段隐藏，只显示审批记录。

**操作步骤**：

1. 在表单配置中，选择目标节点；
2. 找到字段权限配置区域；
3. 对每个字段设置权限：

| 权限类型 | 说明 |
|---|---|
| VISIBLE | 可见但不可编辑 |
| EDITABLE | 可见且可编辑 |
| HIDDEN | 完全隐藏 |
| REQUIRED | 必填（通常与 EDITABLE 配合使用） |

4. 保存配置。

## 9. 变量管理

### 9.1 变量作用域

| 作用域 | 说明 | 使用场景 |
|---|---|---|
| 分类级变量 | 属于某个流程分类，该分类下所有流程可用 | 跨流程通用的变量，如申请人部门、申请日期 |
| 流程级变量 | 仅属于当前流程定义 | 当前流程专用变量，如请假天数、报销金额 |

### 9.2 定义变量

**操作步骤**：

1. 在流程定义列表中，点击目标流程的【变量】按钮；或在分类管理中进入变量配置；
2. 点击【新增】；
3. 填写变量信息：
   - **变量编码**：唯一标识，用于表达式中，如 `leaveDays`；
   - **变量名称**：显示名称，如"请假天数"；
   - **变量类型**：字符串、数字、日期、布尔等；
   - **默认值**：可选；
   - **说明**：可选。
4. 点击【确定】保存。

**注意事项**：

- 同一作用域内变量编码不能重复；
- 变量编码建议使用英文驼峰或下划线命名。

### 9.3 在条件表达式中使用变量

定义变量后，可在连线条件表达式中直接使用：

```
${leaveDays > 3 && amount <= 5000}
```

启动流程时传入对应变量值即可。

## 10. 流程分类管理

### 10.1 使用场景

当流程定义数量较多时，使用分类进行分组管理，如：

- 行政类（请假、报销、用印）
- 人事类（入职、转正、离职）
- 财务类（付款、借款、预算）

### 10.2 新增分类

**操作步骤**：

1. 在流程定义页面左侧分类树上方，点击【新增分类】；
2. 填写分类信息：
   - **分类名称**：如"行政流程"；
   - **分类编码**：如 `admin`；
   - **上级分类**：支持树形结构，可选；
   - **排序**：数字越小越靠前；
   - **状态**：启用 / 停用。
3. 点击【确定】保存。

### 10.3 编辑 / 删除分类

**操作步骤**：

1. 在分类树上右键点击目标分类；
2. 选择【编辑】或【删除】；
3. 确认操作。

**注意事项**：

- 删除分类前，请先将该分类下的流程定义移动到其他分类，或确认可以一并处理。

## 11. 启动流程

### 11.1 从界面启动

**前提条件**：流程定义已发布且状态为启用。

**操作步骤**：

1. 进入对应的流程发起入口（通常在业务菜单或流程中心）；
2. 选择要启动的流程；
3. 填写表单内容；
4. 点击【提交】启动流程。

### 11.2 指定首节点办理人

**使用场景**：第一个审批节点是"部门经理审批"，但系统无法自动确定具体是谁，需要在启动时指定。

**操作方式**：

启动流程时，传入流程变量 `approvalAssignee`，值为用户 ID，例如 `1`。

### 11.3 通过 API 启动

**接口地址**：

```
POST /api/v1/process/instances/start
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| processDefinitionKey | string | 是 | 流程 Key，如 `leave_apply` |
| starter | string | 是 | 发起人用户标识 |
| businessKey | string | 否 | 业务单据编号 |
| formData | object | 否 | 表单数据 |
| variables | object | 否 | 流程变量，可包含 `approvalAssignee` |

**请求示例**：

```json
{
  "processDefinitionKey": "leave_apply",
  "starter": "1",
  "businessKey": "LEAVE20260627001",
  "formData": {
    "leaveType": "annual",
    "leaveDays": 5,
    "reason": "年假"
  },
  "variables": {
    "approvalAssignee": "2"
  }
}
```

**响应示例**：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "instanceId": "10001",
    "status": "running"
  }
}
```

## 12. 常见问题与最佳实践

### 12.1 流程发布失败怎么办？

发布失败通常是 BPMN XML 校验不通过，请检查：

1. 画布中是否有开始事件和结束事件；
2. 所有用户任务是否都配置了办理人或候选人；
3. 排他网关的分支条件是否完整，是否有默认分支；
4. 流程中是否存在孤立的未连接节点。

### 12.2 办理人不生效怎么办？

1. 检查节点是否配置了办理人、候选用户、候选角色或候选部门；
2. 检查角色/部门下是否有成员；
3. 检查是否使用了表达式，但表达式变量未正确传入；
4. 检查会签/或签的集合变量是否正确设置。

### 12.3 条件分支不生效怎么办？

1. 确认变量已在流程变量中定义；
2. 确认启动时传入了正确的变量值；
3. 检查表达式语法，特别是字符串比较和逻辑运算符；
4. 排他网关必须有一条默认分支。

### 12.4 版本管理最佳实践

1. 重大流程变更时升级主版本号（如 1.0 → 2.0）；
2. 小调整升级次版本号（如 1.0 → 1.1）；
3. 发布前先在测试环境验证；
4. 保留历史版本以便追溯和回滚；
5. 停用旧版本前确认没有依赖它的业务在运行。

### 12.5 命名规范建议

| 对象 | 建议 |
|---|---|
| 流程 Key | 英文小写+下划线，如 `leave_apply`、`expense_reimbursement` |
| 节点名称 | 中文，简洁明确，如"部门经理审批" |
| 变量编码 | 英文驼峰或下划线，如 `leaveDays`、`apply_user_id` |
| 分类编码 | 英文小写，如 `admin`、`hr`、`finance` |
