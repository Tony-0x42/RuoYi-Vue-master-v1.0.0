# 系统国际化（i18n）设计文档

## 1. 背景与目标

当前 `ruoyi-ui` 前端大量存在硬编码中文，且无 i18n 基础设施；后端虽有 Spring i18n 骨架，但实际使用范围很小。为满足多语言运营需求，本设计为系统引入可扩展的国际化能力。

**目标：**
- 支持 **中文（zh-CN）/ 英文（en-US）** 切换。
- 为后续法语、日语、阿拉伯语等更多语言预留扩展接口（新语言只需新增 locale 文件并在配置中注册）。
- 优先完成前端通用能力，再按模块渐进式替换文案。

## 2. 范围与阶段

### 第一阶段（本计划实现）

- 前端 i18n 基础设施搭建：`vue-i18n@8`、`Element UI` 语言包联动、语言切换组件。
- 提取并翻译**公共文案**：
  - 按钮（搜索/重置/新增/修改/删除/导出/导入/确定/取消等）
  - 表格操作列
  - 全局提示（request.js、modal.js、errorCode.js）
  - 布局（Navbar、TagsView、Breadcrumb、锁屏、个人中心）
  - 登录页
- 完成 **system/user** 模块作为示例，展示页面级 i18n 改造范式。
- 后端最小改动：
  - 用户表增加/复用 `lang` 字段，保存语言偏好。
  - 登录接口返回当前用户语言偏好。
  - 菜单接口保持现有逻辑（菜单名仍由数据库维护），前端通过 `route.name` 或 `menuId` 做 key 映射；若后台返回 `i18nKey` 则优先使用。

### 第二阶段（后续迭代）

- 按模块（system、monitor、tool、bpm 等）逐批替换 `.vue` 和 `.js` 中的硬编码中文。
- 后端 controller/service/异常信息逐步迁移到 Spring `MessageUtils.message(...)`。
- 引入 CLI 脚本扫描未翻译的硬编码中文，生成待翻译清单。

## 3. 术语

- **Locale**：语言区域标识，如 `zh-CN`、`en-US`。
- **Message Key**：翻译键，命名空间形式，如 `common.search`、`user.form.userName`。
- **Fallback Locale**：当前语言缺失某 key 时回退的语言，固定为 `zh-CN`。

## 4. 方案对比

### 方案 A：前端优先、模块渐进（推荐）

- 前端引入 `vue-i18n@8`，按模块拆分 locale 文件。
- 先做基础设施 + 公共文案 + 1 个示例模块，其余页面逐步替换。
- 后端仅做用户语言偏好持久化和登录返回。

**优点**：风险低、可快速看到效果、不影响业务连续性、扩展性好。  
**缺点**：完全覆盖所有页面需要多轮迭代。

### 方案 B：一次性全量替换

- 一次性扫描所有 ~169 个前端文件和 ~380 个后端文件，全部替换为 key。

**优点**：改造彻底。  
**缺点**：改动量巨大、测试风险高、容易遗漏和回归、无法快速交付。

### 方案 C：仅关键页面做英文版

- 只对登录页、错误页等少量页面写死英文，不做 i18n 架构。

**优点**：改动最小。  
**缺点**：不可扩展、维护困难、违反“为之后适配更多语言留口子”的要求。

## 5. 推荐方案详细设计

### 5.1 技术选型

- **前端库**：`vue-i18n@8.x`（Vue 2 兼容版本）。
- **Element UI 语言包**：`element-ui/lib/locale/lang/zh-CN`、`element-ui/lib/locale/lang/en`。
- **语言持久化**：Vuex store + `localStorage` 双写；登录后从用户信息同步。
- **语言切换组件**：`LangSelect` 下拉，放在 `Navbar` 右侧。

### 5.2 目录结构

```
ruoyi-ui/src
├── i18n
│   ├── index.js              # i18n 实例、语言注册、动态设置
│   ├── locales
│   │   ├── index.js          # 导出所有语言包
│   │   ├── zh-CN.js          # 中文
│   │   ├── en-US.js          # 英文
│   │   └── modules           # 按模块拆分（可选，第一阶段可合并到 zh-CN/en-US）
│   │       ├── common.js
│   │       ├── layout.js
│   │       ├── login.js
│   │       ├── system.js
│   │       └── ...
│   └── helper.js             # $t 封装、key 缺失兜底
├── components/LangSelect     # 语言切换下拉组件
└── store/modules/app.js      # 增加 locale 状态
```

### 5.3 i18n 实例初始化

```js
// ruoyi-ui/src/i18n/index.js
import Vue from 'vue'
import VueI18n from 'vue-i18n'
import messages from './locales'

Vue.use(VueI18n)

const fallbackLocale = 'zh-CN'
const savedLocale = localStorage.getItem('ruoyi-locale')

const i18n = new VueI18n({
  locale: savedLocale || fallbackLocale,
  fallbackLocale,
  silentFallbackWarn: true,
  messages
})

export function setLocale(lang) {
  i18n.locale = lang
  localStorage.setItem('ruoyi-locale', lang)
  document.querySelector('html').setAttribute('lang', lang)
}

export default i18n
```

### 5.4 Element UI 联动

```js
// ruoyi-ui/src/main.js
import ElementLocale from 'element-ui/lib/locale'
import i18n from './i18n'

ElementLocale.i18n((key, value) => i18n.t(key, value))
```

### 5.5 Key 命名规范

采用 `模块.页面.元素` 三段式，全小写驼峰：

- `common.search` — 搜索按钮
- `common.reset` — 重置按钮
- `common.add` — 新增按钮
- `common.edit` — 修改按钮
- `common.delete` — 删除按钮
- `common.export` — 导出按钮
- `common.import` — 导入按钮
- `common.operation` — 操作列
- `user.title` — 用户管理页面标题
- `user.form.userName` — 用户名称表单项
- `user.table.userName` — 用户名称表格列
- `login.usernamePlaceholder` — 登录账号占位符

### 5.6 语言切换组件

```vue
<!-- ruoyi-ui/src/components/LangSelect/index.vue -->
<template>
  <el-dropdown trigger="click" @command="handleSetLanguage">
    <div>
      <svg-icon icon-class="language" />
    </div>
    <el-dropdown-menu slot="dropdown">
      <el-dropdown-item command="zh-CN" :disabled="language==='zh-CN'">中文</el-dropdown-item>
      <el-dropdown-item command="en-US" :disabled="language==='en-US'">English</el-dropdown-item>
    </el-dropdown-menu>
  </el-dropdown>
</template>
```

切换时调用 `setLocale`，并刷新页面或重新加载当前路由（Vue 2 下部分 Element UI 组件需要重新挂载才能完全刷新语言）。

### 5.7 后端最小支持

1. **用户表扩展**：在 `sys_user` 中增加 `lang` 字段，默认 `zh-CN`。
2. **登录返回**：`SysLoginService.login()` 成功后把 `lang` 写入响应或存入 token 载荷。
3. **获取用户信息**：`/getInfo` 返回用户 `lang`。
4. **更新偏好**：新增或复用 `/profile/update` 接口接收 `lang`。
5. **菜单名**：
   - 短期：菜单接口仍返回中文 `menuName`，前端通过 `route.name` 在 locale 文件中查找 key（如 `menu.system.user`）。
   - 长期：数据库 `sys_menu` 增加 `i18n_key` 字段，后端直接返回 key，前端统一 `$t(menu.menuName)`。

### 5.8 公共文案提取清单（第一阶段必须覆盖）

| 区域 | 文件 | 示例 Key |
|---|---|---|
| 路由标题 | router/index.js | `route.dashboard`、`route.systemUser` |
| 登录页 | views/login.vue | `login.title`、`login.username`、`login.password` |
| 全局请求 | utils/request.js | `error.timeout`、`error.network`、`error.unauthorized` |
| 弹窗 | plugins/modal.js | `modal.title`、`modal.confirm` |
| 错误码 | utils/errorCode.js | `errorCode.401`、`errorCode.default` |
| 工具栏 | components/RightToolbar | `toolbar.searchHide`、`toolbar.refresh`、`toolbar.column` |
| 分页 | components/Pagination | `pagination.total` |
| 通用按钮 | 多处 | `common.search`、`common.reset`、`common.add` 等 |
| 用户管理 | views/system/user/index.vue | `user.title`、`user.form.userName`、`user.table.userName` |

## 6. 实施策略

1. **基础设施**：安装 `vue-i18n@8`，创建 `src/i18n`，改造 `main.js`、store、添加 `LangSelect`。
2. **公共文案**：提取公共 key 到 `locales/common.js` / `locales/layout.js` / `locales/login.js`。
3. **示例模块**：改造 `system/user` 全部页面（index、authRole 等）。
4. **后端最小改动**：用户表加字段、登录返回 lang、提供更新接口。
5. **验证**：中英文切换、刷新后保持语言、build 通过。
6. **文档与脚本**：编写迁移指南，提供扫描硬编码中文的脚本。

## 7. 验收标准

- [ ] 前端能正常切换中/英文，切换后刷新页面语言保持一致。
- [ ] Element UI 组件（日期选择器、分页、弹窗按钮等）随语言切换。
- [ ] 公共文案（按钮、登录、错误提示、布局）完成中英双语。
- [ ] `system/user` 模块所有用户可见文案完成中英双语。
- [ ] 后端用户表支持保存语言偏好，登录/获取用户信息返回该字段。
- [ ] 新增语言只需在 `src/i18n/locales/` 下新增一个文件并在 `index.js` 注册，无需改业务代码。
- [ ] `npm run build:prod` 通过。

## 8. 风险与注意事项

1. **Vue 2 生命周期**：语言切换后，部分自定义组件可能需要 `key` 强制重绘才能更新。
2. **后端文案**：第一阶段不处理后端硬编码中文，后端错误信息在英文环境下仍可能返回中文。
3. **菜单翻译**：动态菜单来自数据库，短期通过前端 key 映射，需要维护 `menu.*` key。
4. **图片/静态资源**：带中文的图片需要单独准备多语言版本。
5. **文案遗漏**：后续需要扫描脚本持续发现未翻译的硬编码中文。
